# Email testing

*Mailpit (free, open source, self-hosted) traps every outgoing email locally with a web UI and REST API - the standard for automated tests and CI. Mailinator (free, public inboxes) is fine for a quick manual signup check, but its inbox is PUBLIC - never send anything sensitive there.*

> Verifying a signup, password reset, or order-confirmation flow means completing a step you don't
> actually control: reading a real email that arrives somewhere. Email-testing tools solve this by
> trapping outgoing mail before it ever reaches the real internet — Mailpit catches it locally in a
> dev/CI environment with a clean web UI, while Mailinator gives you disposable public inboxes for
> quick manual checks. Same underlying need, two genuinely different tools depending on what you're
> actually testing.

> **In real life**
>
> A post office's internal sorting room doesn't deliver mail to the outside world at all — every
> piece gets caught, sorted into labeled compartments, and held for inspection right there, visible to
> anyone who walks in and looks. That's precisely how a mail trap behaves: real messages, generated by
> a real app, captured and displayed before they ever reach an actual mailbox — and in Mailinator's
> case, visible to literally anyone who knows which "compartment" (address) to check.

**email testing tools**: Mailpit is a free, open-source local SMTP server (Go-based) that captures every outgoing email from a dev/staging environment in a private web UI plus a REST API, without ever sending real mail. Mailinator is a free hosted service offering public, shared, disposable inboxes at @mailinator.com - convenient for quick manual signup tests, but the inbox is PUBLIC (visible to anyone who knows the address) and its free API tier caps at roughly 100-200 reads/day. Both are verified alive and maintained in 2026 - use Mailpit for automated/CI work, Mailinator for quick disposable manual checks only.

## Two different tools for two different jobs

- **Mailpit** — runs locally (or in CI), catches every email your app tries to send during
  development/testing, displays them in a private web UI, and exposes a REST API a test script can
  query directly to extract a verification link or reset token automatically.
- **Mailinator** — a public, hosted, shared inbox service. Type any address ending in
  `@mailinator.com` and check its inbox instantly — no signup, no setup. Fast for a one-off manual
  check; genuinely PUBLIC, meaning anyone else who tries that same address can read the same mail.
- **The dividing line**: does this need to be private (your own app's real dev/CI environment,
  sensitive tokens involved) or is it a quick, disposable, non-sensitive manual check? That answers
  which tool to reach for.

> **Tip**
>
> For automated tests specifically, Mailpit's REST API is the real payoff: a test script can send the
> signup request, poll Mailpit's API for the resulting email, extract the verification link/token
> programmatically, and complete the flow end-to-end with zero manual steps.

> **Common mistake**
>
> Sending anything containing a real password-reset link, a real verification token tied to actual
> user data, or any genuinely sensitive content to a Mailinator address. Its inboxes are PUBLIC by
> design — anyone who guesses or knows the address can read exactly what you sent. Mailinator is for
> throwaway, non-sensitive manual checks only.

**Completing an email-verification flow with a mail trap**

1. **Configure the app's SMTP settings to point at Mailpit** — Local dev/staging environment - every outgoing email now lands in Mailpit instead of a real inbox.
2. **Trigger the real signup flow** — Fill the form, submit - the app sends its real verification email exactly as it would in production.
3. **Check Mailpit's web UI (or poll its API)** — The email appears immediately, fully rendered, with real headers and body - private, local, nothing left the network.
4. **Extract the verification link/token** — Read it manually for a quick check, or have a test script pull it via Mailpit's REST API for full automation.
5. **Complete the flow using the real extracted token** — Click the link (or hit the URL programmatically) - confirms the FULL flow, not just 'an email was sent'.

The core mechanic — capturing outgoing mail locally and extracting real data from it — is easy to
demonstrate directly:

*Run it - a mail trap catching real app events and extracting a token (Python)*

```python
class MailTrap:
    def __init__(self):
        self.inbox = []

    def send(self, to, subject, body):
        self.inbox.append({"to": to, "subject": subject, "body": body})
        return f"trapped locally, never left the network"

trap = MailTrap()

events = [
    ("new-user@example.com", "Welcome to BuggyShop!", "Click here to verify: /verify?token=abc123"),
    ("new-user@example.com", "Your order has shipped", "Tracking: TRK-88214"),
    ("qa+password-reset@example.com", "Reset your password", "Reset link: /reset?token=xyz789"),
]

print("Triggering three real app events that would normally send email:")
print()
for to, subject, body in events:
    result = trap.send(to, subject, body)
    print(f"  -> {subject!r} to {to}: {result}")

print()
print(f"Mail trap inbox now holds {len(trap.inbox)} messages:")
for msg in trap.inbox:
    print(f"  [{msg['to']}] {msg['subject']}")

print()
verify_email = trap.inbox[0]
token_start = verify_email["body"].find("token=") + len("token=")
extracted_token = verify_email["body"][token_start:]
print(f"Extracted verification token from the trapped email body: {extracted_token}")
print("This is exactly how an automated test completes an email-verification")
print("flow end-to-end - read the trapped message, extract the real token,")
print("use it - without a single real email ever leaving the network.")

# Triggering three real app events that would normally send email:
#
#   -> 'Welcome to BuggyShop!' to new-user@example.com: trapped locally, never left the network
#   -> 'Your order has shipped' to new-user@example.com: trapped locally, never left the network
#   -> 'Reset your password' to qa+password-reset@example.com: trapped locally, never left the network
#
# Mail trap inbox now holds 3 messages:
#   [new-user@example.com] Welcome to BuggyShop!
#   [new-user@example.com] Your order has shipped
#   [qa+password-reset@example.com] Reset your password
#
# Extracted verification token from the trapped email body: abc123
# This is exactly how an automated test completes an email-verification
# flow end-to-end - read the trapped message, extract the real token,
# use it - without a single real email ever leaving the network.
```

Same lesson in Java, laying out the real tradeoff between the two tools this note is built around:

*Run it - Mailinator vs Mailpit, the real tradeoffs (Java)*

```java
import java.util.*;

public class Main {
    public static void main(String[] args) {
        Map<String, String> mailinatorLimits = new LinkedHashMap<>();
        mailinatorLimits.put("inbox_visibility", "PUBLIC - anyone knowing the address can read it");
        mailinatorLimits.put("api_reads_per_day", "~100-200 (free tier)");
        mailinatorLimits.put("domain", "@mailinator.com only");
        mailinatorLimits.put("use_case", "quick manual signup test, not real secrets");

        Map<String, String> mailpitLimits = new LinkedHashMap<>();
        mailpitLimits.put("inbox_visibility", "PRIVATE - local to your dev/staging environment");
        mailpitLimits.put("api_reads_per_day", "unlimited (self-hosted, open source)");
        mailpitLimits.put("domain", "any - it's your own app's real 'to' address");
        mailpitLimits.put("use_case", "automated test fixtures, CI pipelines, real dev work");

        System.out.println("Mailinator vs Mailpit - same job, different tradeoffs:");
        System.out.println();
        System.out.println("Mailinator (public, hosted, quick manual checks):");
        for (Map.Entry<String, String> e : mailinatorLimits.entrySet()) {
            System.out.printf("  %-20s %s%n", e.getKey(), e.getValue());
        }

        System.out.println();
        System.out.println("Mailpit (private, self-hosted, real dev/CI use):");
        for (Map.Entry<String, String> e : mailpitLimits.entrySet()) {
            System.out.printf("  %-20s %s%n", e.getKey(), e.getValue());
        }

        System.out.println();
        System.out.println("Mailinator's PUBLIC inbox is the critical detail: never send");
        System.out.println("anything containing a real password reset link or sensitive");
        System.out.println("token there - anyone who guesses the address can read it too.");
    }
}

     api_reads_per_day    unlimited (self-hosted, open source)
     domain               any - it's your own app's real 'to' address
     use_case             automated test fixtures, CI pipelines, real dev work

   Mailinator's PUBLIC inbox is the critical detail: never send
```

### Your first time: Your mission: complete one email flow end-to-end with a mail trap

- [ ] Set up Mailpit locally (a single binary or Docker container, free) — Follow mailpit.axllent.org's quick-start - it listens on an SMTP port and serves a web UI on another.
- [ ] Point BuggyShop's local dev environment's SMTP config at Mailpit — Check the project's environment configuration for where the mail server address is set.
- [ ] Trigger a real signup or password-reset flow — Submit the form as you normally would - the resulting email should appear in Mailpit's UI within seconds.
- [ ] Open the trapped email and extract the real link/token manually — Confirm it's a genuine, working link - not a placeholder - by actually completing the flow with it.
- [ ] Separately, try a quick check with a Mailinator address — For a NON-sensitive test only - confirm you understand why you would never do this for anything containing a real token.

You've completed a full email-dependent user flow end-to-end, and directly experienced why the
public/private distinction between these two tools matters.

- **No email appears in Mailpit after triggering the flow.**
  Confirm the app's SMTP host/port configuration actually points at Mailpit's listening address - a common cause is an environment variable still pointing at a real mail provider, or Mailpit not running at all. Check Mailpit's own logs for connection attempts.
- **A test script polling Mailpit's API can't find the expected email.**
  Add a short retry/poll loop rather than checking once immediately - email delivery, even locally, isn't always instantaneous, and a single immediate check can race ahead of the message actually landing.
- **A Mailinator inbox shows emails from OTHER people's tests mixed in with yours.**
  This is expected, not a bug - the inbox is genuinely public and shared. Use a sufficiently unique, hard-to-guess address (not just 'test' or 'qa') to reduce collisions, but never rely on it staying private.
- **You need to verify actual EMAIL RENDERING (does it look right in a real client), not just that it arrived.**
  Mail traps confirm content and delivery but don't test rendering across real email clients (Gmail, Outlook, Apple Mail all render HTML email differently) - that requires a dedicated email-rendering preview tool, a separate concern from this note's scope.

### Where to check

- **Mailpit's own web UI and logs** — the first place to check whether an email was actually sent/received and exactly what it contained.
- **The app's SMTP configuration** (environment variables, config files) — the most common root cause when an expected email never appears.
- **Mailpit's REST API documentation** — for building the poll-and-extract pattern into an automated test script.
- **The exact recipient address used** — confirm test scripts and manual checks are looking at the SAME address the app actually sent to; a typo here produces a false "email never arrived" report.

### Worked example: an automated test that completes signup without any manual email checking

1. Task: write an automated test verifying that clicking a signup email's verification link actually
   activates the new account.
2. The test submits the signup form via the app's real API, generating a real user record and
   triggering a real verification email — captured by Mailpit instead of sent externally.
3. The script polls Mailpit's REST API for a message matching the test's specific recipient address,
   with a short retry loop (checking every 500ms, up to 10 seconds) to handle any delivery delay.
4. Once found, the script parses the email body with a regex extracting the verification URL,
   then makes a real HTTP request to that exact URL — completing the verification step exactly as
   a real user clicking the email would.
5. The test then confirms the account's status flipped to "verified" via the app's own API — a
   genuinely complete, end-to-end automated test of an email-dependent flow, with zero manual
   inbox-checking anywhere in the pipeline.

**Quiz.** A tester needs to test a password-reset flow that sends a reset link containing a real, single-use token. Which tool should they use, and why?

- [ ] Mailinator, because it requires no setup and is faster for a one-off manual check
- [x] Mailpit, because the reset token is sensitive (single-use, tied to a real account) and Mailinator's inboxes are PUBLIC - anyone who knows or guesses the address could read and potentially use the token before the tester does
- [ ] Either tool works equally well here since both just display email content the same way
- [ ] Neither - password reset flows containing real tokens cannot be tested without emailing a real personal inbox

*This note's central distinction is exactly this: Mailinator's convenience comes at the cost of a genuinely PUBLIC inbox, which is disqualifying for anything containing a real, sensitive, single-use token - a password reset link is precisely the case this note's Callout warns against sending there. Mailpit's private, local capture is the correct tool specifically because the content is sensitive. Option one prioritizes convenience over the real security implication described in this note. Option three ignores the entire public/private distinction this note is built around - it is not a difference without consequence. Option four is unnecessarily defeatist; a private local mail trap (Mailpit) solves exactly this problem without ever risking real personal data or a real inbox.*

- **Mailpit vs Mailinator — the core distinction** — Mailpit: free, open-source, self-hosted, PRIVATE local capture with a REST API - for automated tests/CI. Mailinator: free, hosted, PUBLIC shared inboxes - fast for quick, non-sensitive manual checks only.
- **Why Mailinator is disqualified for sensitive content** — Its inboxes are genuinely public - anyone who knows or guesses the address can read the same mail. Never send a real password-reset link or sensitive token there.
- **The automated-test pattern email trapping enables** — Trigger the real flow -> poll the trap's API for the resulting email -> extract the real link/token programmatically -> complete the flow with it - a fully automated, no-manual-checking end-to-end test.
- **Why a poll/retry loop matters when checking for a trapped email** — Even local email delivery isn't always instantaneous - checking once immediately can race ahead of the message actually landing, producing a false 'email never arrived' result.
- **What a mail trap does NOT test** — Actual rendering across real email clients (Gmail/Outlook/Apple Mail render HTML differently) - mail traps confirm content and delivery, not visual rendering; that needs a separate rendering-preview tool.
- **Why using a unique Mailinator address still isn't real privacy** — It reduces accidental collisions with other people's tests, but the inbox remains genuinely public by design - uniqueness is not the same as privacy.
